- (note) source: raw/initial-idea.md — HESD workshop digital attendance for 100-150 students per cohort
- (direction) pilot focuses on core attendance flow; defer SSO, academic affairs, school-wide policy, face recognition, native app
- (decision) QR token is short-lived multi-use session token (30s), bound to a workshop session — not one-time for entire class
- (decision) one-successful-check-in rule applies per student per workshop session
- (decision) fraud posture: layered checks reduce cheating, not absolute anti-fraud
- (constraint) scale target 100-150 students per workshop cohort
- (constraint) check-in channel is mobile web, not native app in pilot
- (constraint) login required before check-in completes
- (constraint) GPS location must fall within configured valid range for check-in to succeed
- (constraint) UI follows Neobrutalism design system per docs/ui-ux/design-system/
- (assumption) documentation output language is English per project config despite Vietnamese source idea
- (assumption) implementation stack (framework, database, auth provider) not yet selected — spec stays stack-agnostic
- (question) what GPS geofence radius or boundary definition applies per workshop venue?
- (question) how are student accounts provisioned and authenticated in pilot (email/password, invite list, other)?
- (question) how is the student roster loaded — manual entry, CSV import, or external source?
- (question) are admin and instructor distinct roles with different permissions, or interchangeable in pilot?
- (capability) CAP-1: admin or instructor can create and configure a workshop session including venue/geofence and roster binding
- (capability) CAP-2: system displays a dynamic QR code for an active session that rotates every 30 seconds
- (capability) CAP-3: student can check in via mobile web after scanning the session QR and signing in
- (capability) CAP-4: system validates check-in against QR expiry, roster membership, prior check-in, and GPS before recording attendance
- (capability) CAP-5: instructor or admin can view realtime attendance status for an active session
- (capability) CAP-6: instructor or admin can manually mark or correct attendance when exceptions occur
- (capability) CAP-7: system records an audit log of check-in attempts including failed and anomalous attempts
- (capability) CAP-8: admin can manage the student roster used for workshop attendance
- (capability) CAP-9: instructor or admin can export session attendance as CSV after the workshop
